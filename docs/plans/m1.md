### Day 2-3 — JCS parity harness (BLOCKING)

**Vector architecture (updated 2026-04-20):** spec repo now uses a `inputs/ + generated/ + generate-vectors.mjs` layout. Our harness consumes **only `generated/`** — machine-verified expected values, never hand-authored. This replaces the earlier flat `jcs-parity/*.json` layout that caused the Week 0 parity divergence.

- [ ] Install deps: `pnpm add -w -D vitest` + `pnpm add @filen/rfc8785` in `@soa-harness/core`
- [ ] Create `packages/core/src/jcs.ts` — thin wrapper around `@filen/rfc8785` with type-safe exports
- [ ] **Verify spec-repo generator has been run** — `../soa-harness=specification/test-vectors/jcs-parity/generated/` MUST contain `floats.json`, `integers.json`, `strings.json`, `nested.json`. If `generated/` is empty or any case reports `libraries_agree: false`, STOP — either the spec-repo generator hasn't been run against the pinned commit yet, or there is a genuine library divergence that requires RFC 8785 escalation. In that case, flag on `STATUS.md` for spec-repo session attention.
- [ ] Create `packages/core/test/parity/ts-vs-go.test.ts`:
  - Consume ONLY `../soa-harness=specification/test-vectors/jcs-parity/generated/*.json` (pinned spec commit)
  - For each case with `libraries_agree: true`: run `@filen/rfc8785.canonicalize(case.input)`, assert output bytes === `case.expected_canonical`
  - If any case has `libraries_agree: false` or is missing: fail the test with a pointer to the spec-repo divergence log. Do NOT attempt to fix it here — that's a spec-repo escalation
  - Note: unlike the earlier design, we do NOT shell out to Go from this test. The cross-language comparison happens in the spec-repo generator; our test just verifies our side agrees with the recorded verified output.
- [ ] Run it. Green means Agent Card signing work (Week 1) is unblocked.

## M1 Weeks 1-6 — Components

Each component gets roughly 3-4 working days. Shallow implementation, not deep polish.

### Week 1 — Trust bootstrap + Agent Card

- [ ] `packages/runner/src/bootstrap/` — reads `initial-trust.json`, validates against `schemas/initial-trust.schema.json`, selects channel (SDK-pin for M1, others deferred)
- [ ] `packages/runner/src/card/` — Fastify handler serving `/.well-known/agent-card.json` + `.jws`, detached JWS via `jose` (EdDSA), ETag, `Cache-Control: max-age=300`, JCS via `@soa-harness/core`
- [ ] Integration test: serve a card, verify externally, round-trip passes
- [ ] **Test ID gates:** SV-CARD-01 (spec), SV-SIGN-01 (JWS format)

### Week 2 — StreamEvent emitter

- [ ] `packages/runner/src/stream/` — Fastify SSE endpoint at `/stream/v1/:session_id`, emits closed-enum StreamEvents validated against `stream-event-payloads.schema.json`
- [ ] Unknown type rejected with `ui.gateway-config-invalid` equivalent (flagged for Gateway later; Runner-layer returns schema error)
- [ ] Integration test: connect SSE consumer, verify 5 events round-trip with correct payload types

### Week 3 — Tool Registry + Permission resolver

- [ ] `packages/runner/src/registry/` — static JSON-backed registry (`tools.json`) with `risk_class` + `default_control` per tool; M1 scope, not dynamic MCP registration
- [ ] `packages/runner/src/permission/` — three-axis resolver (capability × control × handler-type), tighten-only override, consumes registry
- [ ] `packages/runner/src/audit/` — JSONL hash-chained audit sink, WAL pattern (`audit/pending/<uuid>` → fsync → append to `permissions.log`), replay on startup (shallow; crash-recovery is M2)
- [ ] **Test ID gate:** SV-PERM-01 (resolver behavior end-to-end)

### Week 4 — PDA verify + CRL cache

- [ ] `packages/runner/src/attestation/` — verifies PDA-JWS against `canonical-decision.schema.json`, consumes spec test vector `test-vectors/permission-prompt/pda.jws` as golden
- [ ] `packages/runner/src/crl/` — three-state cache: `fresh | stale-but-valid | expired`. Only `expired` fails closed. Mock CRL endpoint for integration tests (real CRL fetching is M2/M3 scope)

### Week 5 — Hooks + probes

- [ ] `packages/runner/src/hook/` — pre/post tool hooks, stdin JSON contract, exit-code routing, 5s/10s timeouts
- [ ] `packages/runner/src/probes/` — `/health` (always 200 if process alive), `/ready` (200 only when bootstrap + audit + CRL all green)
- [ ] **Test ID gate:** SV-BOOT-01 (bootstrap + ready probe integration), HR-12, HR-14 (regression tests)

### Week 6 — `create-soa-agent` + 90-second demo

- [ ] `packages/create-soa-agent/src/cli.ts` — interactive CLI (`--name`, `--demo`)
- [ ] `templates/runner-starter/` — working minimal Runner consuming `@soa-harness/runner`
- [ ] `templates/runner-starter/examples/` — one example tool, `AGENTS.md`, demo Agent Card fixture, one PreToolUse hook, one signed PermissionDecision vector
