## Remaining Week 3 day 3+ tasks (ONLY the genuinely unshipped items)

### T-01 — §10.5.3 `GET /audit/records` paginated endpoint `[NEW spec requirement]`

- [ ] Endpoint: `GET /audit/records?after=<record_id>&limit=<n>`
- [ ] Response schema: `schemas/audit-records-response.schema.json` (spec commit `e8b4e9e`)
- [ ] Bearer scope: `audit:read` (existing session bearer carries this); 60 rpm; TLS 1.3 on public listener; loopback plain allowed
- [ ] Pagination semantics: `after` → `next_after` → `has_more`; default `limit=100`, max `1000`; records returned in chain order (earliest first)
- [ ] Not-a-side-effect: reads MUST NOT write a meta-record, MUST NOT emit a StreamEvent
- [ ] Unit tests (seven): empty-log GENESIS page; single-record page; multi-page traversal; `after` with unknown id → 404; `after` with invalid bearer scope → 403; rate-limit exhaustion → 429 (use a deterministic in-test clock + loop to exhaust budget); not-a-side-effect (read twice, assert tail unchanged between reads)

### T-02 — §10.3.2 `POST /permissions/decisions` endpoint `[NEW spec requirement — unblocks audit-record accumulation]`

- [ ] Endpoint: `POST /permissions/decisions` with body `{tool, session_id, args_digest, pda?}`
- [ ] Request schema: `schemas/permission-decision-request.schema.json` (spec commit `80680cd`)
- [ ] Response schema: `schemas/permission-decision-response.schema.json` (spec commit `80680cd`)
- [ ] Bearer scope: `permissions:decide:<session_id>` — **new scope class, distinct from `permissions:resolve:<session_id>`**. Session bootstrap (§12.6) grants `permissions:decide:<session_id>` only when the request body carries `request_decide_scope: true`.
- [ ] Forgery resistance (normative MUST): endpoint computes decision INTERNALLY from §10.3 steps 1–4 + the Tool Registry entry; ignores any client-supplied decision override; PDA `canonical-decision.decision` mismatching the resolver output → `403 reason=pda-decision-mismatch`
- [ ] Side effects: append exactly one audit record per successful call; `response.audit_this_hash` equals new tail hash; StreamEvent emission deferred to M3 (§14 not shipped yet)
- [ ] PDA handling: required when resolver output is `Prompt`; verified against `security.trustAnchors`; cryptographic failure → 201 decision=Deny handler_accepted=false reason=pda-verify-failed AND audit row IS written (the attempt is auditable)
- [ ] Rate limit: 30 rpm per bearer
- [ ] Unit tests (nine): 201 happy path (AutoAllow tool, no PDA), 201 with valid PDA for Prompt tool, 201 with invalid PDA (handler_accepted=false but audit row written), 403 decide scope missing, 403 pda-decision-mismatch, 403 malformed PDA, 429 rate limit, 503 pre-boot, forgery-resistance (decision matches `/permissions/resolve` output for same inputs)

### T-03 — §12.6 bootstrap grants `permissions:decide` when requested `[EXTEND existing POST /sessions]`

- [ ] Add optional `request_decide_scope: true` field to the POST /sessions request schema locally (spec's published schema only documents informative scopes; session_bearer scope list is an impl concern)
- [ ] When present + accepted, the returned session_bearer carries `permissions:decide:<session_id>` scope in addition to `permissions:resolve:<session_id>` and `audit:read`
- [ ] Bootstrap bearer issuer (for local conformance runs, `SOA_RUNNER_BOOTSTRAP_BEARER`) MAY accept or refuse the scope; for M1 conformance accept unconditionally on loopback
- [ ] Unit tests (four): request_decide_scope omitted → bearer has resolve+audit only (decide→403); present + accepted → bearer has all three; scope bookkeeping round-trips through bearer-hash map
