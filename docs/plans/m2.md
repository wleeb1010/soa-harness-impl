### Week 2 — Audit-sink + crash-test

**M2-T4 — Audit-sink three-state degradation (§10.5.1)** — pairs with M2-T4b
- [ ] Three-state machine per §10.5.1: `healthy | degraded-buffering | unreachable-halt`
- [ ] In `degraded-buffering`: actually write records to `/audit/pending/` (fsync-backed), emit `AuditSinkDegraded` StreamEvent once per transition, `/ready` stays 200
- [ ] In `unreachable-halt`: actually refuse Mutating/Destructive with `PermissionDenied reason=audit-sink-unreachable`; permit ReadOnly; emit `AuditSinkUnreachable`; `/ready` flips 503 reason=audit-sink-unreachable
- [ ] On recovery: flush buffer in order + chain-verify + emit `AuditSinkRecovered` → healthy
- [ ] Accept `SOA_RUNNER_AUDIT_SINK_FAILURE_MODE=<healthy|degraded-buffering|unreachable-halt>` env var (§12.5.2) — drives CONCRETE side effects (actual buffer writes, actual refusals) per L-28 F-10 clarification
- [ ] **Fresh boot with env var set emits exactly one matching `AuditSink*` event** per L-28 F-13 clarification (fresh process = implicit `healthy` prior → explicit-named state transition)
- [ ] Production guard: refuse startup with env set + non-loopback bind
- [ ] Tests (10): healthy→degraded transition + actual buffer writes; degraded→unreachable + actual Mutating refusal; unreachable→healthy on recovery; ReadOnly permitted in unreachable-halt; Mutating rejected with audit-sink-unreachable; buffer persistence across restart; chain integrity after flush; env-var test hook drives each state; production-guard refuses non-loopback with env; /ready flips correctly
